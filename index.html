Android Material Design component

Android Material Design component

Google在Android 5.0后为了规范目前纷乱的设计推出了Material Design设计，作为Android的开发人员，紧跟老大的脚步是很重要的。 这里就简单介绍一些常用的Material Design的新增组件。

  • toolbar

toolbar是为了替代规范App Bar而推出的新组件，可以设置返回图标，主副标题，logo，更多按钮，也可以把toolbar当做viewGroup来使用在其中包含自己的布局。
不过需要注意的一点是，toolbar的高度为默认的高度，如果改变这个高度会使自带的返回图标，标题，logo等位置错位。google这么做估计是为了统一规范吧。如过不想使用默认的高度，就自己定义里面的布局吧。

  • AppBarLayout

    这个组件可以把多个布局组合起来，使之成为一个统一的AppBar,其目的是为了配合后面要介绍的CoordinatorLayout 和 CollapsingLayout的使用，来达到滑动的目的。另外它还提供了OnOffsetChangedListener接口来获取AppBarLayout滑动的状态

  • CoordinatorLayout

    这个是Material Design里重量级的新增控件。 在不了解之前，你可以把它视为一个加强版的frameLayout，里面的子view也是一层层堆叠上去的，也提供了诸如android:layout_gravity=”bottom|right”参数来控制view的位置。

这里要注意两点

1.Behavior是Android新出的Design库里新增的布局概念。Behavior只有是CoordinatorLayout的直接子View才有意义。可以为任何View添加一个Behavior。Behavior是一系列回调。让你有机会以非侵入的为View添加动态的依赖布局，和处理父布局(CoordinatorLayout)滑动手势的机会。其中官方提供的app:layout_behavior=”@string/appbar_scrolling_view_behavior”
可以使该view在AppBarLayout之下。如果需要可以自定义behavior.

2.可以通过设置子View,添加app:layout_scrollFlags属性来控制,滚动时候的表现.其中有4种Flag的类型（后两种只有CollapsingLayout才能设置）.

· scroll: this flag should be set for all views that want to scroll off the screen - for views that do not use this flag, they’ll remain pinned to the top of the screen

· enterAlways: this flag ensures that any downward scroll will cause this view to become visible, enabling the ‘quick return’ pattern

· enterAlwaysCollapsed: When your view has declared a minHeight and you use this flag, your View will only enter at its minimum height (i.e., ‘collapsed’), only re-expanding to its full height when the scrolling view has reached it’s top.( 向上滚动时收缩View，但可以固定Toolbar一直在上面。)

· exitUntilCollapsed: this flag causes the view to scroll off until it is ‘collapsed’ (its minHeight) before exiting

  • CollapsingToolbarLayout

    作用是提供了一个可以折叠的Toolbar，它继承至FrameLayout，给它设置layout_scrollFlags，它可以控制包含在CollapsingToolbarLayout中的控件(如：ImageView、Toolbar)在响应layout_behavior事件时作出相应的scrollFlags滚动事件

  • expandedTitleMarginStart - 设置扩张时候(还没有收缩时)title向左填充的距离。

Android 图片压缩

Android 图片压缩

对图片压缩在App应用里是非常普遍和常见的。Android程序的性能和内存问题，大部分都和图片紧密相关，而图片的加载在很多情况下很用到Bitmap（位图）这个类。而由于Bitmap自身的特性（将每个像素的属性全部保存在内存中），导致稍有不慎就会创建出一个占用内存非常大的Bitmap对象，从而导致加载过慢，还会有内存溢出的风险。所以，Android程序要做优化，Bitmap的优化是必不可少的一步。

压缩原理：压缩一张图片，我们需要知道这张图片的原始大小，然后根据我们设定的压缩比例进行压缩。主要分三步

  • 获取原始图片的长宽
  • 计算缩放比例
  • 根据计算出的inSampleSize生成Bitmap

1.获取原始图片的长宽

要说明的一下的是，如果直接使用BitmapFactory.decodeResource()方法来从资源文件中读取一张图片会生成一个Bitmap，它会占用一定的内存。但如果再之前使用一个BitmapFactory.Options对象，并把Options的inJustDecodeBounds属性设置为true，会使得decodeResource(Resources res, int id, Options opts)方法就不会生成Bitmap对象，因为opts控制了decodeResource是对图片完全解码还是只返回图片的大小，当是true时仅仅是读取该图片的尺寸和类型信息。

这是源码中对于inJustDecodeBounds的说明，如果设置为true,编译器返回空即不会返回bitmap,但是会允许调用者查询其属性。 然而不会为bitmap分配内存空间。

2.计算缩放比例

例子中压缩后图片期望的宽带定为480，高度设为800，这2个值只是期望的宽度与高度，实际上压缩后的实际宽度也高度会比期望的要大。如果图片的原始高度或者宽带大约我们期望的宽带和高度，我们需要计算出缩放比例的数值。否则就不缩放。heightRatio是图片原始高度与压缩后高度的倍数，widthRatio是图片原始宽度与压缩后宽度的倍数。inSampleSize为heightRatio与widthRatio中最小的那个，inSampleSize就是缩放值。 inSampleSize为1表示宽度和高度不缩放，为2表示压缩后的宽度与高度为原来的1/2，小于1自动默认为1

Difference from dp dip px sp

Difference from dp dip px sp

Android 开发最让人头疼的问题就是适配问题了，各种手机厂商各种型号的手机鱼龙混杂。所以理解好dp,px,sp等度量单位尤为重要，也是开发的基本常识。
首先我们第一个要了解的是desity的概念，屏幕密度，就是一英寸里有多少个像素点，理论上来说密度越高越清晰,单位为ppi(pixels per inch) Android系统定义了四种像素密度：低（120dpi）、中（160dpi）、高（240dpi）和超高（320dpi），它们对应的dp到px的系数分别为0.75、1、1.5和2

  • px(pixels) 像素,没什么好解释的。例如，320*480的屏幕在横向有320个象素，在纵向有480个象素。
  • dp也叫dip（device indenpendent pixels）设备独立像素，dp不会依赖于手机的分辨率，而是依赖于density，它会随着不同手机的密度不同，自动变化为相应的px长度。举个列子，在density=160时，1dp=1px。这是我们把一根线=120px，在160desity下，它为120px,而在320density下，它变为了原来的一半。 而如果我们把线=120dp，在160desity下，它为120px,而在320density下，它会变为240px，使之与前面的长度相同。
  • sp （scaled pixels）要处理字体的大小。与dp类似，但是可以根据用户的字体大小首选项进行缩放。主要用于字体显示best for textsize。由此，根据 google 的建议，TextView 的字号最好使用 sp 做单位，而且查看TextView的源码可知 Android 默认使用 sp 作为字号单位。

OnMeasure():

首先要知道这个OnMeasure方法是干啥的，为什么要重写它。它的作用是控制View在其父容器中显示的大小，重写它是为了定义view在其中的大小。还是从看代码入手：

首先要先明白是谁调用了onMeasure()这个方法。 调用它的是这个view的父视图。widthMeasureSpec和heightMeasureSpec这两个参数由其父视图viewGroup的layout_height,layout_width，padding以及本身的layout_margin,layout_width,layout_height共同决定（其实还与widght权重有关，但是这个比较复杂这里忽略它）。
接着要来说一下widthMeasureSpec的组成结构，它包含了两部分，Mode and Size. 你可以会奇怪为啥一个Int能包含两部分信息。 其实它是一个32位的，高两位用于表示Model，低30位表示Size。 Model由本身view和Viewgroup的layout_width有关，Size与ViewGroup中得Padding,本身的margin,layout_width。
先来说说Model，它有三种状态：

  • exactly精确状态01：MeasureSpec.EXACTLY：view就为本身指定的大小，再xml中定义的为MATCH_PARENT时model对应于EXACTLY，即view的大小由于父容器指定，但是自己可以指定与边界的距离。
  • at_most最大状态00：MeasureSpec.AT_MOST：view的大小不得大于VIewGroup的大小，再xml中定义的为WRAP_CONTENT对应为于AT_MOST，即自己指定的大小。
  • unspecified为指明状态10:MeasureSpec.UNSPECIFIED： view的大小不受限制，当view设置为制定大小时情况对应这种。
    (怕自己理解错误舞蹈观众还是贴出源码里的描述：
    UNSPECIFIED


    The parent has not imposed any constraint on the child. It can be whatever size it wants.
    EXACTLY


    The parent has determined an exact size for the child. The child is going to be given those bounds regardless of how big it wants to be.
    AT_MOST


    The child can be as large as it wants up to the specified size.)

32位的数字解析出来稍微有点麻烦，所以Google官方也提供了MeasureSpec这个类，它里面封装了由父组件传递给该view的高度和宽度。其中还提供了来获取Model和Size的方法——MeasureSpec.getModel和MeasureSpec.getSize。 最后注意的是setMeasuredDimension(width,height)这里的高宽才是最后真真的大小。

在这里顺便提一下一种让listview或者gridview取消滚动栏方法的实现原理
makeMeasureSpec这个方法是传入size和model返回一个MeasureSpec值。还记得前面说的吗，高两位是model，低位是size，这里我们把最大的整形右移两位空出model的值，剩下的size部分就是30位的最大值了，即size为最大；然后我们又让model设置为AT_MOST即wrap_content有多大显示多大,这样view的高度就成了view本身有多大就显示多大，就不会出现自带的滚动条了~

onLayout():

其目的是为了指定视图的显示位置，方法执行的前后顺序是在onMeasure之后，因为视图肯定是只有知道大小的情况下，才能确定怎么摆放。layout()方法接收四个参数，分别代表着左、上、右、下的坐标，当然这个坐标是相对于当前视图的父视图而言的。在view源码里这是个空方法，这是为什么呢？因为onLayout()过程是为了确定视图在布局中所在的位置，而这个操作应该是由布局来完成的，即父视图决定子视图的显示位置。
既然如此，我们来看下ViewGroup中的onLayout()方法是怎么写的吧。

otto

otto是一个开源工程利用“订阅\发布”来供组件之间事件传递的.它本身库很小但是却能很好的减低代码耦合性(decouple)，和EventBus是同样的作用，相较于Android本身的广播机制来说有诸多好处而且方便实用。

otto简单来说由以下两个构成

  • 一个类： Bus类（用于注册类，注销类，发布事件）

1.创建Bus实例时最好使用单例模式（虽然这不是必须的）

2.发布消息，会全局发送出消息。

3.订阅消息，收到发出的消息，如果和消息的对象一致就接受消息并处理

1.通过继承Bus,当post消息时，如果实在主线程就调用本身的方法，如果不是在主线程，就把post操作利用handle传到主线程，在主线程中调用发送消息方法。这样不论是在主线程还是子线程都会跑到主线程去发送消息。避免了上面那个Demo里的问题

2.然后在这个类中保证了Bus只会被创建一次。以后bus的操作就可以用GlobalBus来操作了。需要使用时只需要GlobalBus.register(this);。OnDestory()时只需要GlobalBus.unregister(this);就可以了。更加方便使用了
